# XFS profile coverage audit

A handoff for reviewing the XFS section catalog the fstests flow runs against, to
decide whether the matrix has the coverage we want. The catalog is generated by
`f/fstests/common.py:xfs_catalog_text()` from `XFS_FEATURES` crossed with the
block/sector geometry matrix; it is the single source (the run-form Sections
picker and `render_config` both read it, and `f/fstests/check.flow`'s
`local.config` default is generated from it by `scripts/gen-fstests-config.py`
(drift-guarded in `nix flake check`), so the editable textarea shows the full
matrix without a hand-maintained copy that can drift).
This is a review handoff: read it, then prune or extend the matrix (`XFS_FEATURES`)
as the coverage decision lands.

## How a profile is built

One `XFS_FEATURES` entry (a mkfs/mount feature knob) is crossed with the geometry
matrix: every `block` in `XFS_BLOCK_SIZES` `[512, 1k, 2k, 4k, 8k, 16k, 32k, 64k]`
and every `sector <= block` in `XFS_SECTOR_SIZES` `[512, 1k, 2k, 4k, 8k, 16k,
32k]`. Section name `xfs_[<feature>_]bs<block>_ss<sector>`. Constraints in the
generator (`xfs_profiles_matrix`):

- V5 (crc, the default) needs `block >= 1024`, so the full geometry per feature is
  34 sections.
- `nocrc` (V4) is the only feature that reaches `block 512`, but a V4 fs above the
  guest page size (4096) is unmountable, so it is capped: 10 sections.
- `realtime_reflink` carries `min_block=4096` (rt reflink via metadir needs a large
  block): 29 sections.
- `realtime_rtx2` is only emitted where `2*block >= XFS_MIN_RTEXTSIZE (4096)`, i.e.
  `block >= 2k`: 32 sections; `realtime_rtx4` clears the floor from `block 1k`: 34.

Total today: **649 sections across 20 features.**

## Realtime / reflink rules (source-confirmed)

Verified in xfsprogs `v7.0.1-37` (`mkfs/xfs_mkfs.c:2998`) and Linux `v7.1`
(`fs/xfs/xfs_super.c:1911`):

- reflink is incompatible with a realtime extent size: mkfs errors if reflink is
  set explicitly, and silently drops it (`reflink = false`, no message) if reflink
  is only the V5 default. The kernel refuses to mount such a fs
  (`"reflink not compatible with realtime extent size %u!"`, `-EINVAL`).
- So `realtime_rtx2`/`realtime_rtx4` (which set `-r extsize` with no explicit
  reflink) correctly produce reflink-off rt filesystems; no `-m reflink=0` is
  needed and none is hardcoded.
- reflink ON with a realtime device needs the metadir feature (the rt refcount
  btree is a metadir inode): that is `realtime_reflink` (`-m metadir=1`,
  `min_block=4096`).

## Coverage gaps to review (for "full coverage")

mkfs.xfs exposes these feature knobs (`-m`/`-i`/`-d`/`-n`): `attr, autofsck,
bigtime, crc, exchange, extsize, finobt, ftype, inobtcount, max_atomic_write,
metadir, nrext64, parent, projid32bit, reflink, rmapbt, sparse, version, zoned,
rgcount`. Crossing that with the matrix, the following are NOT exercised today:

- **`exchange`** (exchange-range / atomic content swap) - a V5 feature, no section.
- **`parent`** (parent pointers) - newer V5 feature, no section.
- **`max_atomic_write`** (atomic writes) - no section.
- **`zoned`** (zoned realtime) - needs a zoned device; candidate Tier-3.
- **`rgcount`** (realtime groups / rtgroups) - rt metadata grouping, no section.
- **`projid32bit`**, **`attr=1`**, **`ftype=0`** - legacy/minor toggles, no section.
- **realtime + reflink + extent size**: `realtime_reflink` does not set `-r
  extsize`, so the rt-reflink path is only tested at the default extent size.
- **more rt extent multipliers** (rtx8, rtx16, ...) and **rt extent sizes on the
  metadir+reflink path** are not covered.

Dimensions the matrix does not vary at all (each could be a tier or a knob):

- **mount options** beyond quota (`-o discard`, `dax`, `noalign`, ...).
- **log geometry** (internal vs external log already partly via `logdev`, but no
  log size / log sector size variants).
- **inode size** (`-i size=256/512/1024/2048`), **directory block size** (`-n
  size=`), **AG count / stripe** (`-d agcount=`, `su`/`sw`).

## Notes for whoever extends this

- Add a feature by adding one `XFS_FEATURES` entry; the matrix, the run-form
  Sections picker, and `render_config`'s fallback pick it up automatically. Then
  re-run `python3 scripts/gen-fstests-config.py` to regenerate the `check.flow`
  `local.config` default; the `generated` check (`nix flake check`) enforces it.
- Geometry-dependent knobs (an extent-size multiplier, a `min_block`) are handled
  in `xfs_profiles_matrix`'s per-block loop; follow the `rt_extsize_mult` pattern.
- External-device features set `needs` (`logdev`/`rtdev`); the renderer binds the
  device and emits `USE_EXTERNAL=yes` + `# external=<needs>`.
- Keep the catalog the single source: `check.flow`'s `local.config` default is
  generated from it by `scripts/gen-fstests-config.py`, never hand-edited (a
  hand-maintained copy drifted before, which is what this audit follows up).
